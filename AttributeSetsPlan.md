# План реализации системы формул

Идея реализуема. Ниже — план и границы.

## 1. Что можно сделать

- **Поле «формула» в инспекторе везде** — через свой тип + **PropertyDrawer**: объявляешь, например, `FormulaField` (сериализуемая структура/класс), везде, где она используется, Unity подхватит один и тот же отрисовщик. Custom Editor для всего класса не нужен.
- **Одна статическая функция** — хранится только строка формулы; в рантайме одна статическая функция, например `FormulaEvaluator.Evaluate(formula, Dictionary<string, float>)`, парсит строку и возвращает `float`. Так и делаем.
- **Подсказка по доступным ключам** — в редакторе у поля «формула» можно показывать список имён параметров, но источник нужно задать явно: в edit-time словаря нет, только в рантайме. Вариант: у типа формулы есть опциональная ссылка на объект, который «знает» имена параметров (например, твой `PlayerData` или отдельный asset со списком ключей). PropertyDrawer по этой ссылке получает список имён и рисует подсказку.

## 2. План по частям

### 2.1. Тип «формула» (сериализуемый)
Структура/класс с полями:
- `string formula` — сама формула (например, `"AvailableDistance * 0.5 + Health"`).
- Опционально: ссылка на объект-источник имён параметров (например, `ScriptableObject context` или `PlayerData context`), чтобы в PropertyDrawer показывать подсказку «доступны ключи: …».

### 2.2. PropertyDrawer для этого типа
В папке `Editor`:
- Класс с атрибутом `[CustomPropertyDrawer(typeof(FormulaField))]`.
- В `OnGUI`: 
  - рисуешь строковое поле для формулы (или одно большое текстовое поле);
  - сверху или снизу — подсказка (HelpBox/Label) со списком имён параметров;
  - список имён получаешь из опционального поля «context»: если там asset с списком параметров (например, `PlayerData` с `parameters` или интерфейс `IParameterNamesProvider`), берёшь оттуда имена и выводишь.
- Если context не задан — показываешь нейтральную подсказку («используй имена ключей из твоего Dictionary»).

### 2.3. Источник имён для подсказки
Чтобы подсказка была «из вне» и в формате «как в Dictionary»:
- В `PlayerData` (или в отдельном asset) хранится список имён параметров (у тебя уже есть `parameters` с `NamedFloat.name`).
- Тип формулы хранит необязательную ссылку на этот asset; PropertyDrawer по ссылке вызывает что-то вроде `GetParameterNames()` и рисует их списком. Можно ввести интерфейс, например `IParameterNamesProvider` с методом `string[] GetParameterNames()`, и реализовать его в `PlayerData` — тогда поле формулы можно привязать к любому объекту, который реализует этот интерфейс (и отображать подсказку только когда ссылка задана).

### 2.4. Парсер и одна статическая функция
- Отдельный класс, например `FormulaEvaluator`, со статическим методом:
  - `float Evaluate(string formula, Dictionary<string, float> parameters)`.
- Внутри: парсер выражений (числа, идентификаторы = ключи словаря, `+`, `-`, `*`, `/`, скобки; при желании — вызовы вроде `Max(a,b)`, `Clamp01(x)`).  
Реализация — либо свой простой парсер (рекурсивный спуск, ~50–100 строк), либо готовая библиотека для выражений (например, NCalc), в зависимости от сложности формул.

## 3. Что сложнее

- **Подсказка «везде без настройки»** — чтобы список ключей подставлялся сам, без явной ссылки на asset, нужен глобальный источник (например, статический список имён или «текущий выбранный» asset в редакторе). Это возможно, но менее предсказуемо для геймдизайнера («откуда взялись эти имена?»). Проще и понятнее: у поля формулы есть опциональное поле «Context» (ссылка на PlayerData или другой asset с ключами); если задано — показываем подсказку, если нет — общая фраза.
- **Очень сложные формулы** — если понадобятся циклы, свои функции и т.д., парсер усложнится; для арифметики и простых функций идея не слишком амбициозная.

---

## 4. Краткий порядок шагов

1. Ввести тип `FormulaField` (или `SerializableFormula`) с полями: `string formula`, опционально `Object context` / `PlayerData context`.
2. Реализовать `FormulaEvaluator.Evaluate(string, Dictionary<string, float>)` с простым парсером (или библиотекой).
3. Написать PropertyDrawer для `FormulaField`: поле строки формулы + подсказка из `context` (список имён параметров).
4. При желании — интерфейс `IParameterNamesProvider`, реализация в `PlayerData`, использование в drawer для универсальной подсказки «из вне» в формате «те же ключи, что и в Dictionary».

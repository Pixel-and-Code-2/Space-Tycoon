# Жизненный цикл Monobehaviuor

### 1. Работа в Editor Mode

Многие думают, что скрипты "спят", пока не нажата кнопка Play. Это не совсем так.

#### `OnValidate`

- **Когда вызывается:** Каждый раз, когда вы меняете значение любого поля скрипта в Инспекторе, а также при загрузке сцены или перекомпиляции кода.
- **Зачем нужен:** Для проверки данных. Например, если у вас есть поле `health`, и вы хотите, чтобы оно никогда не было отрицательным:
  ```csharp
  private void OnValidate() {
      health = Mathf.Max(0, health); // Ограничиваем значение прямо в редакторе
  }
  ```
- **Важно:** Не используйте здесь тяжелую логику или создание объектов, так как это замедлит работу самого редактора Unity.

#### `Reset`

- **Когда вызывается:** Только в редакторе, когда вы впервые добавляете компонент на объект или когда нажимаете кнопку "Reset" в контекстном меню компонента.
- **Зачем нужен:** Для установки значений по умолчанию, которые отличаются от стандартных типов (например, автоматический поиск ссылки на соседний компонент через `GetComponent`).

#### `[ExecuteAlways]` и `[ExecuteInEditMode]`

Если над классом поставить этот атрибут, то методы `Update`, `OnEnable` и другие будут работать даже тогда, когда игра **не запущена**. Это полезно для создания инструментов визуализации в редакторе.

### 2. Awake vs Start: Главная битва

Это самый важный момент для стабильности архитектуры.

#### `Awake` (Инициализация "себя")

- Вызывается **сразу** после создания объекта (или загрузки сцены).
- Вызывается, даже если компонент `enabled = false` (но сам GameObject активен).
- **Что делать:**
  - Поиск собственных компонентов: `GetComponent<Rigidbody>()`.
  - Инициализация внутренних списков и переменных.
  - Настройка синглтонов (`Instance = this`).

#### `Start` (Инициализация связей)

- Вызывается на первом кадре, непосредственно перед первым `Update`.
- Вызывается **только если** скрипт включен (`enabled = true`).
- **Что делать:**
  - Взаимодействие с другими объектами. К моменту вызова `Start` у **всех** объектов в сцене уже гарантированно отработал `Awake`.
  - Подписка на события других классов.
  - Передача данных между скриптами.

**Золотое правило:**

> В `Awake` я настраиваю себя. В `Start` я здороваюсь с соседями.

### 3. Порядок выполнения (Execution Order)

Если у вас есть два скрипта, и `Start` одного зависит от данных из `Start` другого — вы получите ошибку, так как порядок вызова `Start` между разными объектами **не определен**.

Для решения этой проблемы:

1.  Используйте связку `Awake` (подготовка данных) -> `Start` (чтение данных).
2.  Или настройте **Script Execution Order** в настройках проекта (`Project Settings -> Script Execution Order`), чтобы один скрипт всегда просыпался раньше другого.

### 4. Подробный разбор цикла обновления

1.  **FixedUpdate:** Вызывается с фиксированным интервалом (по умолчанию 0.02с). Весь физический код (движение через `Rigidbody`, силы) должен быть здесь.
2.  **Update:** Вызывается каждый кадр. Зависит от мощности ПК. Здесь мы ловим ввод игрока (`Input.GetKeyDown`) и считаем таймеры.
3.  **LateUpdate:** Вызывается после всех `Update`. Идеально для следования камеры за игроком. Если игрок передвинулся в `Update`, камера в `LateUpdate` увидит его уже в новой позиции, и не будет "дрожания".

### 5. Завершение жизни

- **OnDisable:** Вызывается, когда вы выключаете скрипт или объект. Хорошее место для отписки от событий.
- **OnDestroy:** Вызывается перед полным удалением объекта. Здесь нужно чистить ресурсы (например, вручную созданные `Mesh` или `RenderTexture`), чтобы не было утечек памяти.
